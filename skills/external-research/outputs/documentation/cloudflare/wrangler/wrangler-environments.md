Cloudflare Wrangler CLI environments allow you to create different configurations for the same Worker application, enabling distinct settings for development, staging, and production, among other use cases[1][2]. These environments are configured within your Worker's Wrangler configuration file, which can be either `wrangler.toml` or `wrangler.jsonc` (Cloudflare recommends `wrangler.jsonc` for new projects)[1][3].

Key aspects of Wrangler environments include:
*   **Configuration** Environments are defined in the configuration file using `[env.<ENV_NAME>]` sections. For example, `[env.dev]` would define a development environment[1][4].
*   **Naming Convention** When an environment is created, Cloudflare effectively creates a new Worker with the name `<top-level-name>-<environment-name>`. For instance, a Worker named `my-worker` with a `dev` environment would deploy as `my-worker-dev`[1].
*   **Usage with Commands** Environments are activated using the `--env` or `-e` flag with Wrangler commands. For example, to develop in a `dev` environment, you would run `npx wrangler dev -e=dev`, and to deploy, `npx wrangler deploy -e=dev`[1][2].
*   **Inheritance** Most configuration keys are inheritable from the top-level configuration. This means that settings defined at the top level of your `wrangler.toml` or `wrangler.jsonc` file will apply to all environments unless explicitly overridden within an environment's section[3].
*   **Non-inheritable Keys** Some keys, such as `bindings` and `environment variables`, are non-inheritable and must be specified individually for each environment where they are needed[1][3].
*   **Environment Variables and Secrets** Wrangler facilitates managing environment-specific variables and secrets. Environment variables can be defined directly in the configuration file under the `vars` key within each environment. For sensitive values like API keys, Wrangler supports environment secrets, which are stored securely and not directly in your configuration file[2][4][5].
