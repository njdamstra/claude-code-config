 cssAnimationPlugin analysis - what it does:
1. Applies CSS classes (animate-active, animate-{preset}) to trigger animations
2. Sets CSS custom properties (--animation-duration, --animation-delay, --animation-easing)
3. Listens for animationstart/end/cancel/iteration events
4. Syncs with environment plugin (pause/resume on battery/fps/reduced motion)
5. Maps preset names to @keyframes (slideUp  slideInUp, fadeIn  fadeIn)

It's designed for components that don't have their own @keyframes - it brings external animation definitions into the element.

- Plugin adds complexity (class management, event listeners, cleanup) for what CSS already does natively

The plugin essentially "wraps" CSS animations to make them controllable via JS API. But theaters already control animations via:
- CSS variables for dynamic values
- animation-play-state for pause/play
- Class toggles for activation

Adding the plugin feels like abstracting something that's already simple.

 Performance comparison:

Current (pure CSS):
- 0 JS execution during animation (GPU handles everything)
- ~50 lines CSS per component
- No event listeners during steady state
- Browser can optimize @keyframes aggressively

With cssAnimationPlugin:
- Initial JS execution to apply classes + setup listeners
- 4 event listeners per animated element (start/end/cancel/iteration)
- WeakMap lookups on every cleanup
- Class add/remove on play/stop (triggers style recalc)
- 370 lines of plugin code included in bundle

For 4 duplicate cards in ShootingPosts: 16 event listeners
For 3 theaters on page: ~50 event listeners total

Not massive overhead, but measurable. Current approach is leaner for continuous animations.

// Consistent pattern everywhere
const anim = useAnimation(el, { cssAnimation: { preset: 'fadeIn' } })

- Children = presentation layer (CSS animations make sense)

 Key insight - Plugin designed for DIFFERENT animation patterns:

cssAnimationPlugin is optimized for:
- One-shot animations (button clicks, page transitions)
- Animations with completion callbacks needed
- Dynamically starting/stopping individual elements
- Reusing global animation presets

Theater animations are:
- Continuous loops or long-running sequences
- Controlled by parent active prop, not individual triggers
- Theater-specific @keyframes, not reusable presets
- Multiple elements animating in sync with stagger

This is a fundamental mismatch! cssAnimationPlugin was designed for UI interactions (modals appearing, toasts sliding in), not complex theater choreography.

The current architecture already found the right pattern: useAnimation for coordination, CSS for execution.

 Alternative solution - Provide/Inject pattern without plugin:

Stage could provide animation control context:
```typescript
// Stage
const animationControl = {
  shouldPause: computed(() => shouldPauseAnimations.value),
  speed: ref(1.0)
}
provide('animationControl', animationControl)

// Children
const control = inject('animationControl')
watch(() => control.shouldPause.value, (pause) => {
  el.style.animationPlayState = pause ? 'paused' : 'running'
})
```

This achieves centralized control WITHOUT cssAnimationPlugin overhead. Pure CSS animations remain, but Stage coordinates pause/resume state.

This is probably the "best of both worlds" - keeps CSS simplicity, adds coordination.

 Final verdict synthesis:

Current architecture is CORRECT for theater use case because:
1. Theaters are continuous/long-running animations, not one-shot interactions
2. Pure CSS performs better for infinite loops (0 JS overhead)
3. Animation co-location with components aids maintainability
4. Each theater has unique @keyframes (not reusable presets)
5. Simple class toggle (:active) sufficient for on/off control

cssAnimationPlugin would ADD value if:
1. Theaters needed programmatic animation lifecycle events
2. Stage required direct pause/resume control over children (but provide/inject solves this)
3. Animation presets needed to be swapped dynamically (they don't)
4. Multiple useAnimation instances justified their overhead (they're not)

Recommendation: Keep current pattern, optionally add provide/inject for centralized pause control if battery optimization becomes priority. cssAnimationPlugin is excellent for UI interactions but overly complex for theater choreography.
